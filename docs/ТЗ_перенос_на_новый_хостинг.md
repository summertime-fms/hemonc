# Техническое задание: перенос сайта hemonc на новый хостинг

---

## 1. Цель и результат

Перенести действующий сайт на 1С-Битрикс с текущего хостинга на новый сервер **без потери данных, без простоя более согласованного окна и с сохранением всех внешних интеграций**.

Критерии приёмки:

- Сайт полностью открывается по рабочему домену на новом сервере (публичная часть + админка `/bitrix/admin/`).
- Все интеграции (1С, CallTouch, Bitrix24, аналитика, короткие ссылки) работают и проверены
- Настроены фаервол, резервное копирование

## 5. Настройка фаервола

### 5.1. Сетевой фаервол (хост)

Использовать `ufw`/`nftables`/`iptables` или облачные Security Groups. Политика по умолчанию — **всё запрещено, разрешаем точечно**.

**Входящий трафик (ingress):**

| Порт     | Протокол | Источник                            | Назначение        |
| -------- | -------- | ----------------------------------- | ----------------- |
| 80/tcp   | HTTP     | `0.0.0.0/0`                         | редирект на HTTPS |
| 443/tcp  | HTTPS    | `0.0.0.0/0`                         | сайт              |
| 22/tcp   | SSH      | только доверенные IP/VPN            | администрирование |
| 3306/tcp | MySQL    | **закрыт извне** (только localhost) | БД                |

**Исходящий трафик (egress) — минимально необходимый:**

- 443 к `laskov.medconsult.ru` (1С), `api.calltouch.ru`, доменам Bitrix24/Метрики/VK.
- 25/465/587 к SMTP-релею (исходящая почта).
- 53 (DNS), 80/443 для обновлений ОС и Битрикс.

Пример `ufw`:

```bash
ufw default deny incoming
ufw default allow outgoing
ufw allow 80/tcp
ufw allow 443/tcp
ufw allow from <ADMIN_IP> to any port 22 proto tcp
ufw enable
```

MySQL слушает только `127.0.0.1` (в `my.cnf`: `bind-address = 127.0.0.1`), порт 3306 наружу не публикуется (в отличие от dev-конфига `docker-compose.yml`, где проброшен `13306:3306` — **на проде так делать нельзя**).

### 5.2. Прикладной уровень (web application)

- Включить и настроить модуль Битрикс `security` (проактивный фильтр WAF), сменить `sid` кэша.
- Закрыть `/bitrix/admin/` дополнительно (Basic-Auth или ограничение по IP на веб-сервере).
- Форсировать HTTPS + HSTS, `Options -Indexes`, скрыть служебные каталоги (`/upload/` — запретить исполнение PHP).
- `fail2ban` на SSH и на попытки брутфорса `/bitrix/admin/`.
- Отключить листинг директорий, запретить прямой доступ к `.settings.php`, `dbconn.php`, `/local/debug/`, `.git/`.
- Регулярные обновления ОС и ядра/модулей Битрикс.

## 6. Резервное копирование

### 6.1. Что бэкапим

1. **База данных** — полный дамп (`mysqldump --single-transaction --routines --triggers --events`).
2. **Файлы** — весь `site/`, особенно `site/upload/` и не версионируемые конфиги (`.settings.php`, `dbconn.php`, `.env`).
3. **Конфигурация сервера** — конфиги веб-сервера, PHP, MySQL, cron, `msmtprc`, правила фаервола.

### 6.2. Расписание и хранение (рекомендация)

| Тип                          | Частота                                         | Хранение | Схема ротации                                    |
| ---------------------------- | ----------------------------------------------- | -------- | ------------------------------------------------ |
| БД (полный дамп)             | ежедневно ночью                                 | 30 дней  | 7 ежедневных + 4 еженедельных + 6 месячных (GFS) |
| Файлы (`upload/` и код)      | ежедневно инкрементально, полный — раз в неделю | 30 дней  | GFS                                              |
| Полный образ/снапшот сервера | еженедельно                                     | 4 недели | —                                                |

- Хранить копии **вне основного сервера** (отдельное хранилище/облако)
- Бэкапы **шифровать** (в них секреты и ПДн пациентов) и ограничить доступ.
- Учитывать требования к ПДн (мед. тематика): хранение и передача — в соответствии с 152-ФЗ, доступ по журналу.
